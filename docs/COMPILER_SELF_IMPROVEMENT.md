# AXIOM Compiler Self-Improvement Analysis

**Date:** 2026-03-25
**Analyzed:** 35,802 lines of Rust across 8 crates

## Critical Finding: llvm.rs is 13,674 lines

`crates/axiom-codegen/src/llvm.rs` contains 338 functions in a single file. This is:
- Difficult to navigate
- Slow to compile incrementally
- Hard to review changes
- Impossible to test in isolation

## Other Compiler Issues Found

### 1. axiom-mir crate is empty (1 line)
The MIR (Mid-level IR) crate has been a stub since day 1. Either implement it or remove it from the workspace to avoid confusion.

### 2. Two TODO markers remain
- `main.rs:507` — `--emit` for unknown levels says "TODO"
- `axiom-mir/src/lib.rs:1` — the entire crate is TODO

### 3. Compilation warnings
`try_const_eval_body` and `eval_const_expr` are unused (dead code warnings). These were superseded by the recursive const evaluator but never removed.

### 4. axiom_rt.c is 1,843 lines
The C runtime has grown from a tiny file to nearly 2K lines. It should be split into:
- `axiom_rt_io.c` — file I/O, clock, args
- `axiom_rt_threads.c` — threads, atomics, mutexes, job system
- `axiom_rt_coroutines.c` — fiber/ucontext coroutines
- `axiom_rt_renderer.c` — Win32 software renderer
- `axiom_rt_audio.c` — audio builtins
- `axiom_rt_input.c` — input tracking

### 5. Benchmark code quality varies
Some benchmarks were generated by AI and have:
- Inconsistent coding style
- Missing `@constraint` annotations
- Large stack arrays that should be heap (found by verifier)
- No `@strategy` blocks (can't be auto-optimized)

## Improvement Opportunities for Performance

### Compiler Compilation Speed
The axiom-codegen crate takes the longest to compile because llvm.rs is huge. Splitting it into modules would enable:
- Parallel compilation of modules within the crate
- Faster incremental rebuilds (change one builtin, don't recompile all 338 functions)

### Better Code Generation Patterns
Review the emit_* functions for patterns that generate suboptimal LLVM IR:
- Multiple redundant loads from the same alloca (mem2reg handles this, but cleaner IR = faster optimization)
- Unnecessary sext/trunc for type mismatches (the binary op coercion could be smarter)
- Large match blocks that could use lookup tables

### New Benchmark Ideas from Compiler Analysis
The compiler itself suggests benchmarks:
- **Lexer throughput**: tokenize a large .axm file, measure tokens/second
- **Parser throughput**: parse a complex .axm file, measure AST nodes/second
- **Codegen throughput**: generate LLVM IR for a complex program, measure lines/second
- **Full pipeline**: .axm → binary compilation time for various program sizes

These would be self-referential benchmarks — AXIOM measuring its own compiler's performance.
